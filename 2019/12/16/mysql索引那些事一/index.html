索引结构

索引存在的目的是为了提高查询效率的. 就像书的目录一样的. 但是实现索引的方式却有很多种. 有三种比较常见的. 哈希表, 有序数组, 搜索树.

哈希表

哈希表是一种kv结构. 输入待查找的值key, 就可以得到对应的value. 实现的思路也很简单. 通过hash函数算出固定长度的整数和桶的长度求摸. 算出index桶的位置. 依据鸟巢原理可知. 必然存在重复的index. 我们拉出一条链表存储相同的hash值的情况. 当然解决哈希冲突的方式很多种, 除了拉链法还是有线性寻址法. ThreadLocalMap就是这种方案.

对于等值查询. hash表示非常高效的. 但是对于范围查询. 就只能全表扫描了. so 哈希索引适用于等值查询. 所以在很多nosql数据库很常见.

有序数组

有序数组是非常适合等值和范围查询的. 但是如果只看查询效率, 数组是非常合适的数据结构. 数组的弊端就是对于静态数据很好用. 但是对于动态数据的操作, 在查询同时,数组的元素会因为插入和删除引起数据搬移. 这开销是很难接受的.

搜索树

平衡的二叉树 能够保证O(logn)时间复杂度. 索引不存要存储在内存中, 还要存储磁盘中, 那么一个100w个节点的平衡二叉树数据. 树高就是20左右. 就意味着20次磁盘I/O, 磁盘IO是毫秒级别的. 一次大约10ms左右, 使用二叉树. 一次查询就要2s. 这查询是太慢了.

所以我们需要降低树的高度. 那么可以增加树的节点数. 那么这个数的节点数n到底数多大呢? 以InnoDB为例. 数据存储是区 段 页 行. 也就是说一次磁盘IO读取一页的数据. innoDB中默认的页大小是16k = 16384b大小. 拿主键索引举例, bigint是8b, 页指针是6b, 16384/14=1170. innodb是一个1170叉树. innodb中的索引树结构也叫做b+树. 和b树的区别就是b树非叶子节点也是存储数据的. 而b+树非叶子节点只存储索引值和页指针.叶子节点是按照索引顺序的存储的双向链表, 这在范围查询的时候区别就出来了. b树范围查询需要通过中序遍历完成. b+树只要确定了沿着链表顺序遍历即可.

innodb的索引结构

在innodb中. 每个索引就是一颗b+树. 分聚集索引和非聚集索引. 聚集索引值的是主键索引. 不仅如此. innodb在存储数据的时候也是按照主键索引组织数据的. 而且自增的主键索引的好处还有在插入数据是维护索引树的时候.
因为每页的大小是固定的. 索引在插入的插入和删除的时候, 会伴随着页的分裂和合并. 这会使得页的空间利用率下降. 而自增的主键索引就不同了. 只要在页里追加即可. 避免了页的分裂和合并.

很多业务字段的主键. 是不能保证顺序写入的. 会增加数据的插入成本.

非聚集索引的叶子节点存储的是索引及其主键索引的指针.

显然. 主键长度越小，普通索引的叶子节点就越小，普通索引占用的空间也就越小。

在典型的kv场景中, 业务字段作为主键是一种优化手段, 否则会要搜索两颗索引树的. 比如 只有一个索引. 该索引必须是唯一索引的场景. 可以将唯一索引设置为主键. 从搜索两颗树变成一颗树.

尽量使用主键查询原则.

回表

我们知道通过非聚集索引查询, 会搜索两颗索引树. 因为分聚集索引的叶子节点存储了主键索引. 而这种会主键索引树查询的过程就叫做回表.

执行的select * from T where k between 3 and 5 的流程是什么样的呢?

这里相当于 查询 3<=k>=5的数据

  1. 在k索引树上找到k=3的数据, 拿到ID=300;
  2. 去ID树搜索ID=300的数据. 添加到结果集
  3. 继续在k树中搜索>=3的数据. 找到k=5的. 拿到ID=500
  4. 去ID树搜索ID=500的数据, 将数据添加到结果集;
  5. 继续在k树上搜索到k=6的数据. 不符合条件. 循环结束

在这个过程中，回到主键索引树搜索的过程，我们称为回表。可以看到，这个查询过程读了 k 索引树的 3 条记录（步骤 1、3 和 5），回表了两次（步骤 2 和 4）.

覆盖索引

如果执行的语句是 select ID from T where k between 3 and 5，这时只需要查 ID 的值，而 ID 的值已经在 k 索引树上了，因此可以直接提供查询结果，不需要回表。也就是说，在这个查询里面，索引 k 已经“覆盖了”我们的查询需求，我们称为覆盖索引。

由于覆盖索引可以减少树的搜索次数，显著提升查询性能，所以使用覆盖索引是一个常用的性能优化手段。

在考虑创建索引的时候, 每当为索引添加一个字段的时候都意味着索引变大. 导致每页存储的指针变小. 从而影响查询效率. 但我们要不要再建立(id_card,name)索引. 是通过id_card查询name, 使用覆盖索引. 减少回表次数.优化查询呢. 这个要结合业务来判定了.

最左前缀原则

最左前缀是指, 对于(name,age)的索引, 早查询中必须包含name条件. 这样才能应用到这个索引. 例如 where name=’张%’, 这个是可以应用的索引的. 所以联合索引的顺序是非常重要的, 如果通过调整索引顺序能够减少维护一个索引. 那么这个优化是有必要的,

根据最左前缀原则. 针对 name= and age= 的查询是可以应用索引的. 但是单独的age=是无法应用索引的. 如果查询age也是高频的. 那么需要给age单独建立索引. 或者考虑name,age的索引是否调整顺序.

索引下推

select * from tuser where name like '张%' and age=10 and ismale=1; 这个sql在5.6之前的查询, 在索引name_age上.去找到姓张的人. 然后依次回表查询找到age=10的和ismale=1的. 这样是要很多次回表,

在5.6之后. 可以在name_age索引书上, 过滤掉age!=10的, 从而减少了回表次数.

普通索引和唯一索引的选择问题

比如select name from CUser where id_card = 'xxxxxxxyyyyyyzzzzz'; 针对这个sql. 我们都知道身份证号是不会重复, 那么是对id_card字段建立普通索引还是唯一索引的. 对于判定是否有下一个数据, 对于引擎来说就是一次寻址操作.

  1. 查询 在存储结构上两个索引是没有区别的. 再查询效率上也是相差无几的. 区别就是 普通索引在查询的时候要判断是否还有其他数据. 而唯一索引直接就范围了, 因为唯一嘛. 这能查多少效率呢. 微乎其微吧
  2. 更新操作. 更新过程中呢, 更新一条数据, 当引擎发现这个数据页内存中, 在不影响数据一致性的前提下，InooDB 会将这些更新操作缓存在 change buffer 中，这样就不需要从磁盘中读入这个数据页了。在下次查询需要访问这个数据页的时候，将数据页读入内存，然后执行 change buffer 中与这个页有关的操作。通过这种方式就能保证这个数据逻辑的正确性。这个change buffer也是有持久化操作的. 持久化时机是, 查询这个数据的时候回触发merge操作, 数据库关闭时也会触发merge操作, 如果更新的时候能用应用到change buffer减少磁盘I/O, 无疑会提高更新效率.

因为唯一索引的唯一约束. 必须回到磁盘中才会知道数据是否唯一. 所以无法使用change buffer. 而普通索引没有这个约束. 可以使用change buffer.

这时 唯一索引和普通索引的差距就很大了, 因为针对每条更新操作, 都会涉及减少一次磁盘I/O的操作. 要知道磁盘io是数据最大的开销之一.

但是change buffer页不是万才能的. 我们知道触发merge是在更新后查询. 针对一些业务场景. 比如更新数据后, 立即伴随一次查询. 这种情况change buffer只会增加开销. 起不到优化作用. 因为增加了chenge buffer的维护. 如果我们的业务是每次更新都会伴随着查询. 应该关闭change buffer.

change buffer和redo log的区别

  1. redo log是事务持久化的重要组件. innodb的redo log是固定大小的, 从头开始写记录write pos. 写到末尾在从头开始写. 可以理解为循环数组. checkpoint是当期要擦除的位置. 也是往后推移并且循环的，擦除记录前要把记录更新到数据文件。 当write pos追上checkpoint的时候. 说明redo log满了. 这个时候就不能再执行更新操作了, 得停下来先擦掉一些记录，把 checkpoint 推进一下。

innodb通过redo log保证了更改数据的安全. 即便是数据库宕机, 依然可以通过redo log恢复之前提交的数据, 保证不丢失.

redo log减少的随机写io, 变成顺序IO, change buffer减少的是随机读IO.

redo log 和binlog

redo是innodb独有的, 保存的物理日志, 记录数据页的修改, 循环写入. 有用完的时候.
binlog是追加写. 追加写意味着一个文件写完,会切换到另外一个文件继续写, 保存的是逻辑日志, 是mysql service层实现的.

为什么mysql会选错索引

选错索引是因为优化器选择索引的目的，是找到一个最优的执行方案，并用最小的代价去执行语句。在数据库里面，扫描行数是影响执行代价的因素之一。扫描的行数越少，意味着访问磁盘数据的次数越少，消耗的CPU资源越少.

针对大字符串怎么建立索引

如果前缀区分度高的化. 取前缀建立索引.

或者可以反转 例如身份证在简历索引是可以先字符串反转.

或者依据字符串生成哈希值. 查询是通过hash && equals进行查询, 缺点是不能范围查询.

假设有这样一张表:

需要执行sqlselect city,name,age from t where city='杭州' order by name limit 1000 ;

Using filesort使用到的排序算法

Using filesort说明使用文件排序. 会为每个线程开辟一块大小固定的sort_buffer进行在内存排序.

这个sort_buffer是有大小限制的. 当需要用到的字段大小之和小于max_length_for_sort_data值时. 使用全字段排序. 大于时使用rowid排序 这里的rowId值的是行记录的唯一表示. innodb引擎的主键.

两种排序的区别是 后者会回表.

当需要排序的数据量大于sort_buffer_size时. 会借助磁盘临时文件帮助排序.

全字段排序

依据执行计划 explan返回的结果 Extra 这个字段中的“Using filesort”表示的就是需要排序，MySQL 会给每个线程分配一块内存用于排序，称为 sort_buffer。

执行流程是

  1. 初始化sort_buffer, 确定放入 name city age三个字段.
  2. 从索引city中拿到满足=’杭州’的第一个记录,拿到主键ID.
  3. 到主键索引取出整行记录. 取出 name city age的值 存入sort_buffer.
  4. 从索引city取下一条记录的ID.
  5. 重复3.4直到. 满足city=’杭州’的记录用完,
  6. 对sort_buffer中的数据进行快速排序.
  7. 按照排序结果返回前1000条数据

整个上面的过程是就是全字段排序. 是发生在内存中的. 也可能要借助外部排序. 取决于sort_buffer_size的大小 如果排序所需的内存小于这个值 在内存中排序. 如果大于则要借助磁盘临时文件完成,

外部排序的算法是基于sort_buffer_size大小的. 例如我们要排序900m大小的数据. 但是内存给的buffer只有100m, 那么我们就需要借助外部磁盘临时文件进行排序, 分别对每个文件进行排序. 然后归并结果集即可.

rowid排序

不管是内存排序还是借助外部磁盘临时文件排序. 都有一个问题. 如果排序返回的字段过多的话. 那么sort_buffer中存储的行数就会越少, 要使用外部磁盘文件的数量也就会越多. 排序的性能就会很差. 所以单行很大.全字段排序的效果就不是很好, 这里就引入了rowid排序.

max_length_for_sort_data 这属性决定了当返回的字段加起来超过这个值的时候, 就会使用rowid排序. 不超过就会用全字段排序.

city、name、age 这三个字段的定义总长度是 36，我把 max_length_for_sort_data 设置为 16，我们再来看看计算过程有什么改变。新的算法放入 sort_buffer 的字段，只有要排序的列（即 name 字段）和主键 id。

那么排序流程就变成了了:

  1. 初始化sort_buffer. 放入name和id两个字段
  2. 从索引city中拿到满足=’杭州’的第一个记录,拿到主键ID.
  3. 到主键索引中取出整行数据, 取出name和id连个字段, 放入sort_buffer.
  4. 从索引city取出下一条记录的ID
  5. 重复3.4直到满足city=’杭州’的数据用完,
  6. 对sort_buffer中的数据按照name字段进行快速排序
  7. 遍历排序结果中前1000条数据. 并按照主键ID回表查询, 取出city name age.返回给客户端.

这里可以看出rowid排序增加了回表操作, 这涉及到了磁盘IO
如果 MySQL 实在是担心排序内存太小，会影响排序效率，才会采用 rowid 排序算法，这样排序过程中一次可以排序更多行，但是需要再回到原表去取数据。如果 MySQL 认为内存足够大，会优先选择全字段排序，把需要的字段都放到 sort_buffer 中，这样排序后就会直接从内存里面返回查询结果了，不用再回到原表去取数据。
这也就体现了 MySQL 的一个设计思想：如果内存够，就要多利用内存，尽量减少磁盘访问。对于 InnoDB 表来说，rowid 排序会要求回表多造成磁盘读，因此不会被优先选择。

并不是所有order by语句都会涉及额外的排序, 如果在索引上取出的数据天然有序. 就不会需要排序的. 例如将index改为alter table t add index city_user(city, name); 这样在执行上面的sql语句. name字段在city相等的情况下, 就是天然递增的.

整个查询的流程就变为了:

  1. 通过city,name索引. 找到第一个满足的 拿到主键ID
  2. 会主键索引树, 取出整行记录. 将city name age三个字段的值. 作为结果集的一部分直接返回.
  3. 从city,name索引 继续取得下一条记录
  4. 重复2,3步骤, 直到查到第1000条, 或者第一条不满足city=’杭州’的记录为止

针对这个查询的sql, 我们还可以优化他, 让查询避免回表. 就是覆盖索引. 建立这样的索引alter table t add index city_user_age(city, name, age);.

随机排序

mysql> select word from words order by rand() limit 3;

针对这样的sql 怎么优化呢. 在数量很小的情况. 这样的sql是不会出问题的. 当数量很大的时候.

依据explain执行计划. 在Extra 字段显示 Using temporary，表示的是需要使用临时表；Using filesort，表示的是需要执行排序操作。 这里的意思是需要使用临时表进行额外的排序. 这里的filesort使用的全文排序还是rowid排序呢 答案是rowid. 因为临时表示内存中的表, 不需要涉及磁盘IO, 使用rowId排序能够跑正sort_buffer容纳更多的行信息. 减少外部磁盘临时文件的使用.

这里的排序流程就变成了:

  1. 构建临时表, 引擎是memory.表里有两个字段, 1 varchar类型W. 2. double类型的即为R. 没有建立索引
  2. 从words表中 按照主键顺序取出所有word值. 假如10000行, 对于每个word, 调用rand函数 生成0到1之间的随机小数. 并将随机小数和word分别放入临时表的R和W字段中 扫描行数10000
  3. 临时表中的10000条记录, 接下来要在临时表上按照R进行排序.
  4. 构建一个sort_buffer buffer中有两个字段 1 double类型 2 整形信息
  5. 从内存临时表中一行一行的取出R和位置信息(innodb中位置信息表示主键. memory引擎中表示下标. 可以理解为数组) 分别添加到buffer中的两个字段中. 扫描行数10000 -> 20000
  6. 在sort_buffer中 依据R字段进行排序.
  7. 排序完成后 取出前三个. 去临时表中取出word值. 返回给客户端. 20000 -> 20003

这里的order by rand() limit 3使用了内存临时表, 内存临时表排序的时候使用了rowid排序

并不是所有的临时表都是内存临时表, 这里使用参数控制的. tmp_table_size设置内存临时表大小 默认16m 超了 就变成磁盘临时表了 internal_tmp_disk_storage_engine 指定磁盘临时表的引擎 默认innoDB.

这里还有一个问题 就是sort_buffer使用的排序是前面提到的归并排序 还是 新的排序算法. 因为这里的排序比较条数. `limit 33值很小. 也就是说10000条记录我们只要前三个. 如果使用归并, 就意味着要对10000条数据全排序, 得到10000条从小到大的有序数据. 这很浪费. 我们只需要前三条. 对于所有的求top问题. 我们知道要使用堆解决. 优先队列. mysql也不例外. 这里使用了堆排序. 维护一个3元素个大小大顶堆. 遍历10000条记录, 例如原R和堆顶元素T, 如果R < T `` 我们删除T. 将R添加到堆中. 在遍历完成后. 堆中的三个算是就是最小的三个元素.
