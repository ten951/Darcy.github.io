网络那些事儿

发表于 2020-06-05 | 分类于 NETWORK

network

试图去解释清楚客户端请求服务端涉及到关键问题

浏览器

生成HTTP请求

打开chrome浏览器, 输入网址 https://www.google.com/?hl=zh-cn https是协议 当然还有其他的比如 http/ftp/file/mailto等. 协议顾名思义就是数据传输的格式.

  1. 浏览器解析URL. http://web服务器名/目录名/文件名 例如: http://www.lab.glasscom.com/dir1/file1.html 目的是知道访问的目标是哪里.
  2. http协议. 定义了客户端和服务器之间交互消息内容和步骤. 其基本思路是. 客户端会向服务器发送请求消息. 请求消息中包含的内容是”对什么”和进行怎样的操作. “对什么”就是资源地址. “进行怎样的操作”就是方法. 例如GET POST等. 收到客户端的请求后. 服务器端会对内容进行解析. 依据url和方法扥这些要求来完成自己工作, 然后就结果放在响应消息中. 其中状态码表示服务器执行的结果.
  3. 对URL进行解析之后,确定了web服务器和文件名. 要依据这些信息生成HTTP消息. 这是有严格的格式规定的. 请求 {请求行(GET URL HTTP/1.1), 请求头(Content-Type: text/plain;charset=UTF-8), 消息体(传输的数据,编码依据header头中的设置).} 响应 {状态行(HTTP/1.1 200 OK) 响应头(Content-Type: application/json; charset=utf-8) 响应正文(传输的数据)}
  4. 生成HTTP消息之后, 需要委托给操作系统将消息发送给WEB服务器. 但此时的消息不具备发送到网络的能力. 想要将消息发送到网络中. 需要TCP/IP协议支持. 所以我们需要知道web服务器对应的IP地址. 这就用到了DNS服务.

向DNS服务器查询WEB服务器的IP地址

为什么要ip地址和服务器域名并用. 统一使用ip地址或者域名不就可以了么?

  1. 统一使用ip是可以的. 但是出于商业目的. 人类记住字符的能力要高于数字. 比如www.google.com很容易记住.但是背后的ip地址就很难,和电话号码一样. 出于传播
  2. 处于效率考虑网络传输时使用ip定位路由器. 因为ip的长度时32bit. 也就是4字节(Byte) 而且域名最短也要十几个字节. 出于效率

DNS查WEB域名IP地址的步骤: 如 http://www.google.com

  1. 查询机器本身的HOSTS
  2. 查询最近的DNS服务器
  3. 查询根域名.
  4. 查询 .com
  5. 查询 .google.com

直到了WEB域名对应的IP地址后. 需要委托协议栈发送消息

委托协议栈发送消息

这里的协议栈指的就是TCP/IP的四层模型(应用层,传输层,网络层,网络接口层). 应用层指的就是http file这些. 可以简单地理解为Data. 这些数据按相应的协议格式封装好数据. 例如HTTP 格式. 将Data 准备好后需要将这些数据发送到目标服务器. 就需要用到传输层协议TCP网络层协议IP等协议参与. 而这些协议非常复杂. 操作系统屏蔽这些复杂性. 例如如何和网卡交互等. 提供了程序组件Socket. 所以委托协议栈发送消息. 可以理解为按照一定顺序调用socket api的过程. 其中寻址(DNS查IP也是其中的功能)就是其中的一部分.

TCP/IP - 协议栈和网卡

TCP/IP协议栈是一系列网络协议的总和. 是构成网络通信的核心骨架. 定义老电子设备如何链接因特网. 以及数据如何在他们直接进行传输. 采用4层结构. 分布是应用层(HTTP FTp).传输层(TCP UDP),网络层(IP ARP)和链路层(以太网 网桥).

网络通信好比快递. 商品外面是一层一层包裹就是各种协议. 协议包含了商品信息, 收获地址. 联系方式等. 然后需要配送车 配送站 快递员, 商品才能最终到达用户手中. 一般情况 快递是不能直达的. 需要先转发到对应的配送站. 然后由配送站再进行派件.

配送车好比物理介质. 配送站好比网关. 快递员就是路由器. 收货地址就是IP. 联系方式就是MAC地址.

这一流程涉及的协议很多. 特别复杂.

Socket是网络应用的基础. 位于应用层. 是操作系统为应用程序员提供的API. 通过它应用程序可以访问传输层协议. 屏蔽了网络协议之间的差异. 屏蔽了系统差异.

委托协议栈发送消息, 可以简单理解为创建Socket的过程. Socket实体就是通信控制信息. 在协议栈内部有一块用于存放控制信息的内存空间. 记录了通信对象的IP地址. 端口号 通信操作的进行状态. 在发送数据时. 需要看Socket中通信对象IP地址和端口号. 在发送数据之后. 需要等待对方返回收到数据的相应信息. 但数据也可能中途丢失. 永远也等不到对方相应的情况. 在等待一定时间后会重新发送丢失数据. 具体需要等待多长数据, 这些都是需要Socket去依据以往的发送相应时间来确定.

创建Socket 并连接.

指定协议类型. 端口号.协议栈会分配所需的内存空间. 然后初始化信息. 创建好套接字之后, 应用程序就会调用connect. 协议栈会将本地的套接字与服务器的套接字进行连接. 套接字刚刚创建的时候并没有存放任何数据. 也不知道通信对象是谁. 即使应用程序要求发送.协议栈也不知道应该发送给谁. 协议栈需要知道目标ip和端口. 这就是connect的目的之一.

调用了connect api. 将ip和端口传递给了协议栈中的tcp模块. TCP模块会与该IP地址对应的对象. 也就是服务器的TCP模块交换控制信息. 这个过程就是三次握手.

  1. 客户端标识位为SYN a 携带本次连接方向上要使用的第一个序列号. ISN a. 自己进入SYN_SEND状态.
  2. 服务端收到消息. 向客户端发送标志位:SYN b 并携带此方向上自己要使用的第一个ISN b. + ACK(ISNa+1) 自己进入SYN_RCVD状态
  3. 客户端收到服务端发送的SYNb和ACK之后. 发送一个ACK给服务端表示自己收到了. ACK(ISNb+1) 自己进入establish状态

三次握手完毕. 连接已经建立. 客户端就可以发送数据了.

当控制流程从connect回到应用程序之后. 接下来就进入数据收发阶段了. 从应用程序调用write将要发送的数据交给协议栈开始. 协议栈不关心数据是什么内容. 在他看来都是一定长度的二进制数据. 协议栈将数据放在内部缓冲区中. 等待应用程序发送下一段数据. 需要累计到一定量时在发送出去. 要累计多少数据. 是依据两个要素的.

  1. 网络包能容纳的数据长度. MTU值.是包含头部的总长度. MSS值是MTU值减去头部长度. 是网络数据包长度. 当累计到MSS值时 发送数据.
  2. 协议栈有个计时器. 当经过一定时间后. 即便没有到MSS大小, 也会发送数据

长度优先. 有效了的利用的网络资源. 但是会有较长的延迟. 要等待缓冲区填满. 时间优先呢. 延迟小,但是利用率低. 因为可能达不到MSS.

HTTP请求的消息一般不会很长 一个网络包能够装下. 遇见较大的数据, 例如表单提交. 长度超过了一个网络包容纳的数据量. 会以MSS长度作为单位进行拆分.
