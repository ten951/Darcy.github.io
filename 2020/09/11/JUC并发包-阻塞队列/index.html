JUC并发包-阻塞队列

发表于 2020-09-11 | 分类于 多线程
字数统计: 3.2k 字 | 阅读时长 ≈ 14 分钟

阻塞队列(BlockingQueue)

在JUC包下提供了几个线程安全的阻塞队列. 例如ArrayBlockingQueue,LinkedBlockingDeque,LinkedBlockingQueue,LinkedTransferQueue,PriorityBlockingQueue,SynchronousQueue.
之所以叫阻塞队列. 意思是在一定条件下可以自己挂起自己. 等待满足条件时, 被唤醒

ArrayBlockingQueue

有界的 基于数组实现的阻塞队列, 在初始化时要确定cacapacity.

实现原理

基于重入锁和两个基于锁的等待队列实现的, 所以有公平和非公平只说. 这里的公平和非公平是指竞争资源时时先来后到还是大家一起抢.谁抢到算谁的.

关键API put(E e)、offer(e, time, unit)和take()、poll(time, unit)

内部实现了一个环形数组. 维护两个index. putIndex和takeIndex及其count记录数组中元素的个数. 基本思路为:

  • 添加元素时putIndex赋值并+1. 判定putIndex如果等于数组长度. putIndex =0; count++; notEmpty.signal();
  • 删除元素时takeIndex赋值null并+1 如果takeIndex等于数组长度. takeIdnex = 0; count–; notFull.signal();

put 添加元素

重入锁加锁. 保证线程安全. 当队列慢的时候. 调用线程在notFull condition队列上等待. 否则enqueue方法入对

take 获取元素

重入锁加锁. 保证线程安全. 当队列空时. 调用线程在notEmpty condition队列上等待. 否则Dequeue方法出队

总结

生产者消费者共用一把锁. 可能会存在瓶颈.

LinkedBlockingQueue

近似有界的阻塞队列. 默认的大小是Integer.MAX.

实现原理

实现方案: 内部数据结构是单链表. 使用两把重入锁和两个condition队列. 实现线程安全. 队列中元素的个数使用原子类AtomicInteger.

添加方法 put和offer

基本流程相似:

  1. putLock.lock()
  2. 检查count和容量capacity比较. 相等容量用尽. 要在notFull.condition上awit的
  3. enqueue数据
  4. count+1. 并和capacity比较. 如果小于. 说明有空间. 唤醒notFull.condition上的等待.
  5. putLock.unLock()
  6. count == 0 说明 存在一个元素. 唤醒消费者去获取数据

区别:

  • put方法是. 没有可用容量的时候.会在condition队列上阻塞. 等待空间可用被唤醒.
  • offer(E e)方法. 容量满了不会阻塞. 直接返回false. 容量不满的时候. 会执行enqueue.
  • offer(E e, long timeout, TimeUnit unit) 容量满了. 会在condition队列上等待timeout.

获取方法 take和poll

基本流程相似:

  1. takeLock.lock
  2. count 等于 0 则在takeLock.condition队列上等待数据来临.被唤醒
  3. dequeue 数据出队 X
  4. count - 1 和 1(因为有head)比较. 队列还有数据. 继续唤醒takeLock.condition中等待的线程.
  5. 数据数量c和capacity相等, 没弄明白

区别

  • take方法会阻塞. 直到有数据
  • poll()方法 队列没数据会立刻返回null. 否则会阻塞等待唤醒
  • poll(time) 方法. 等待数据到来一定时间

总结

生产者消费者使用两把锁控制, 能够得到更好的性能. 但因为默认值大小是Integer.Max.
不能设置公平和非公平策略. 底层使用单链表. 相对于数组实现的ArrayBlackQueue会多消耗内存.

PriorityBlockingQueue

优先级阻塞队列 无界(MAX - 8). 基于堆实现的. so. 元素必须实现comparable接口. 内部使用小顶堆.

这里的大小理解为优先级会好理解一些. 用数字表示就是堆顶是最小的元素

用堆实现优先队列的源码和过程. 我的另一篇博客算法相关总结和梳理有详细讲解

由于是无界队列, 所以添加数据不需要阻塞. 只要使用lock保证临界区线程安全即可. 数据添加完成. 唤醒有可能等待的获取数据的线程. 为了保证空间利用率. 需要动态扩容支持

扩容

上述源码表明了扩容调用和执行.

  1. while(size >= queue.length) 循环保证扩容的原子性.
  2. 这里的unlock 的用途为. 保证添加和获取元素的时候 能够同时扩容
  3. allocationSpinLock标识了 是否有人在扩容. 0 没有 1 存在扩容. CAS的从0改为1. 成功说明没有竞争. 失败说明存在竞争. (case 5)
  4. array 是参数传递进来的原数组. queue是全局实现的那个数组. 如果==比较为true. 说明是自己在扩容,而非另外一个线程完成扩容.
  5. Thread.yield(); 这里我的理解是等待其他线程完成扩容. 这里算是一个小优化, 避免过多的线程在lock上等待

添加 offer

  1. 获取lock
  2. 检查是否需要扩容
  3. 上浮
  4. size + 1
  5. 唤醒可能在等待的消费者线程
  6. 解锁

获取 take poll

  1. 获取lock
  2. 获取堆顶数据E. 获取堆尾数据N, 将N放置堆顶. 然后执行下沉. 恢复堆有序
  3. Size - 1
  4. 检查E == null. true 线程阻塞. 在notEmpty上等待. 被唤醒
  5. 解锁

SynchronousQueue

这是一个非常特殊的阻塞队列. 内部实现有栈和队列两种数据结构.

  1. 入队线程和出队线程必须一一匹配. 否则任意线程到达都会阻塞.
  2. 内部不保存任何元素. 容量为0.数据直接在配对的生产者和消费者线程之间传递，不会将数据缓冲到队列中。
  3. 支持公平和非公平. 非公平使用栈实现. 公平使用队列实现
  4. 基于一种 Dual stack and Dual Queue的无锁算法实现

构造

无论是TransferQueue还是TransferStack. 都继承了Transferer.

栈结构

mode 三个值的含义

  • REQUEST 未配对的消费者. 当线程执行出队操作时. 会创建一个REQUEST类型的SNode item 是空
  • DATA 未配对的生产者. 当线程执行入队操作时. 会创建一个Data类型的SNode. item 非空
  • FULFILLING 表示成功配对的生产者/消费者
核心操作-put/take
  • 情况 1 h==null || h.mode == mode, 队列为null 或者 mode相等表示无法配对. 要执行入队操作
  • 情况 1.1 限时等待的情况.
  • 情况 1.2 构建SNode节点, next指针指向head. 然后CAS的将自己替换为head. 成功后执行awaitFulfill方法阻塞自己 1: 获取当前线程. 2. 自旋 判断s.math是否为null 不为null 说明匹配成功 返回math. 为null 说明需要阻塞. 设置s节点的waiter. 调用locksupport.park阻塞
  • 情况 2.0 h.mode和mode不相等. 且 & FULFILLING =0 . 说明 h.mode和mode 不相等. 存在可以配对的可能性. 尝试去匹配
  • 情况 2.1 标记新head节点3为 (FULFILLING|mode) 相当于打标(11 or 10 ). 这样其他线程 !isFulfilling(h.mode) 一定是false. 获取s.next m m.next mn 匹配s和m. 匹配的过程就是修改m的match为s. 成功弹出两个节点. s和m. 将mn设置为head. 会唤醒s.waiter线程. 线程会在awaitFulfill方法这里被唤醒 进入下一次循环. 发现match != null 返回匹配对象m 获得data.
