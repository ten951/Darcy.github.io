sql优化相关

发表于 2020-10-16 | 分类于 MySql
字数统计: 3.3k 字 | 阅读时长 ≈ 12 分钟

原则篇

  • 最左前缀原则 mysql一直向右直到遇到范围查询(> < between like)就停止匹配. 所以要把等值条件放在前边

  • 尽量选择区分度高的列做索引. 希望能够过滤掉更多的数据. 例如 主键ID就是区分度最高的索引. 扫描一条就可以

  • 用 union all 替换 or.

  • 字段加函数不适用索引

  • 注意字段类型. 类型转化不会使用索引

  • 字段运算不会使用索引 例如a+10 >20 不会用到索引. a > 10 使用索引

  • 使用联合索引的时候 必须包含第一列

  • is null is not null != 这些都应该避免使用

  • order by 中所有的列必须包含在相同索引中保持在索引中的排列顺序. 不能即包含asc也包含desc

  • 使用 exits 或者 not exits 取代 in和not in

  • 清除 不必要的排序. 尽量使用索引自身的有序性. 完成排序

  • 尽量使用join代替子查询

  • 尽量使用 union all 代替 union 因为 union会去重. 占用cpu

    原理篇

count * 的问题

  • MyISAM 引擎把一个表的总行数存在了磁盘上，因此执行 count(*) 的时候会直接返回这个数，效率很高；
  • InnoDB 引擎就麻烦了，它执行 count(*) 的时候，需要把数据一行一行地从引擎里面读出来，然后累积计数. 因为存在MVCC

order by 工作原理

假设有这样一张表:

需要执行sqlselect city,name,age from t where city='杭州' order by name limit 1000 ;

Using filesort使用到的排序算法

Using filesort说明使用文件排序. 会为每个线程开辟一块大小固定的sort_buffer进行在内存排序.

这个sort_buffer是有大小限制的. 当需要用到的字段大小之和小于max_length_for_sort_data值时. 使用全字段排序. 大于时使用rowid排序 这里的rowId值的是行记录的唯一表示. innodb引擎的主键.

两种排序的区别是 后者会回表.

当需要排序的数据量大于sort_buffer_size时. 会借助磁盘临时文件帮助排序.

全字段排序

依据执行计划 explan返回的结果 Extra 这个字段中的“Using filesort”表示的就是需要排序，MySQL 会给每个线程分配一块内存用于排序，称为 sort_buffer。

执行流程是

  1. 初始化sort_buffer, 确定放入 name city age三个字段.
  2. 从索引city中拿到满足=’杭州’的第一个记录,拿到主键ID.
  3. 到主键索引取出整行记录. 取出 name city age的值 存入sort_buffer.
  4. 从索引city取下一条记录的ID.
  5. 重复3.4直到. 满足city=’杭州’的记录用完,
  6. 对sort_buffer中的数据进行快速排序.
  7. 按照排序结果返回前1000条数据

整个上面的过程是就是全字段排序. 是发生在内存中的. 也可能要借助外部排序. 取决于sort_buffer_size的大小 如果排序所需的内存小于这个值 在内存中排序. 如果大于则要借助磁盘临时文件完成,

外部排序的算法是基于sort_buffer_size大小的. 例如我们要排序900m大小的数据. 但是内存给的buffer只有100m, 那么我们就需要借助外部磁盘临时文件进行排序, 分别对每个文件进行排序. 然后归并结果集即可.

rowid排序

不管是内存排序还是借助外部磁盘临时文件排序. 都有一个问题. 如果排序返回的字段过多的话. 那么sort_buffer中存储的行数就会越少, 要使用外部磁盘文件的数量也就会越多. 排序的性能就会很差. 所以单行很大.全字段排序的效果就不是很好, 这里就引入了rowid排序.

max_length_for_sort_data 这属性决定了当返回的字段加起来超过这个值的时候, 就会使用rowid排序. 不超过就会用全字段排序.

city、name、age 这三个字段的定义总长度是 36，我把 max_length_for_sort_data 设置为 16，我们再来看看计算过程有什么改变。新的算法放入 sort_buffer 的字段，只有要排序的列（即 name 字段）和主键 id。

那么排序流程就变成了了:

  1. 初始化sort_buffer. 放入name和id两个字段
  2. 从索引city中拿到满足=’杭州’的第一个记录,拿到主键ID.
  3. 到主键索引中取出整行数据, 取出name和id连个字段, 放入sort_buffer.
  4. 从索引city取出下一条记录的ID
  5. 重复3.4直到满足city=’杭州’的数据用完,
  6. 对sort_buffer中的数据按照name字段进行快速排序
  7. 遍历排序结果中前1000条数据. 并按照主键ID回表查询, 取出city name age.返回给客户端.

这里可以看出rowid排序增加了回表操作, 这涉及到了磁盘IO
如果 MySQL 实在是担心排序内存太小，会影响排序效率，才会采用 rowid 排序算法，这样排序过程中一次可以排序更多行，但是需要再回到原表去取数据。如果 MySQL 认为内存足够大，会优先选择全字段排序，把需要的字段都放到 sort_buffer 中，这样排序后就会直接从内存里面返回查询结果了，不用再回到原表去取数据。
这也就体现了 MySQL 的一个设计思想：如果内存够，就要多利用内存，尽量减少磁盘访问。对于 InnoDB 表来说，rowid 排序会要求回表多造成磁盘读，因此不会被优先选择。

并不是所有order by语句都会涉及额外的排序, 如果在索引上取出的数据天然有序. 就不会需要排序的. 例如将index改为alter table t add index city_user(city, name); 这样在执行上面的sql语句. name字段在city相等的情况下, 就是天然递增的.

整个查询的流程就变为了:

  1. 通过city,name索引. 找到第一个满足的 拿到主键ID
  2. 会主键索引树, 取出整行记录. 将city name age三个字段的值. 作为结果集的一部分直接返回.
  3. 从city,name索引 继续取得下一条记录
  4. 重复2,3步骤, 直到查到第1000条, 或者第一条不满足city=’杭州’的记录为止

针对这个查询的sql, 我们还可以优化他, 让查询避免回表. 就是覆盖索引. 建立这样的索引alter table t add index city_user_age(city, name, age);.

随机排序

mysql> select word from words order by rand() limit 3;

针对这样的sql 怎么优化呢. 在数量很小的情况. 这样的sql是不会出问题的. 当数量很大的时候.

依据explain执行计划. 在Extra 字段显示 Using temporary，表示的是需要使用临时表；Using filesort，表示的是需要执行排序操作。 这里的意思是需要使用临时表进行额外的排序. 这里的filesort使用的全文排序还是rowid排序呢 答案是rowid. 因为临时表示内存中的表, 不需要涉及磁盘IO, 使用rowId排序能够跑正sort_buffer容纳更多的行信息. 减少外部磁盘临时文件的使用.

这里的排序流程就变成了:

  1. 构建临时表, 引擎是memory.表里有两个字段, 1 varchar类型W. 2. double类型的即为R. 没有建立索引
  2. 从words表中 按照主键顺序取出所有word值. 假如10000行, 对于每个word, 调用rand函数 生成0到1之间的随机小数. 并将随机小数和word分别放入临时表的R和W字段中 扫描行数10000
  3. 临时表中的10000条记录, 接下来要在临时表上按照R进行排序.
  4. 构建一个sort_buffer buffer中有两个字段 1 double类型 2 整形信息
  5. 从内存临时表中一行一行的取出R和位置信息(innodb中位置信息表示主键. memory引擎中表示下标. 可以理解为数组) 分别添加到buffer中的两个字段中. 扫描行数10000 -> 20000
  6. 在sort_buffer中 依据R字段进行排序.
  7. 排序完成后 取出前三个. 去临时表中取出word值. 返回给客户端. 20000 -> 20003

这里的order by rand() limit 3使用了内存临时表, 内存临时表排序的时候使用了rowid排序

并不是所有的临时表都是内存临时表, 这里使用参数控制的. tmp_table_size设置内存临时表大小 默认16m 超了 就变成磁盘临时表了 internal_tmp_disk_storage_engine 指定磁盘临时表的引擎 默认innoDB.

这里还有一个问题 就是sort_buffer使用的排序是前面提到的归并排序 还是 新的排序算法. 因为这里的排序比较条数. `limit 33值很小. 也就是说10000条记录我们只要前三个. 如果使用归并, 就意味着要对10000条数据全排序, 得到10000条从小到大的有序数据. 这很浪费. 我们只需要前三条. 对于所有的求top问题. 我们知道要使用堆解决. 优先队列. mysql也不例外. 这里使用了堆排序. 维护一个3元素个大小大顶堆. 遍历10000条记录, 例如原R和堆顶元素T, 如果R < T `` 我们删除T. 将R添加到堆中. 在遍历完成后. 堆中的三个算是就是最小的三个元素.
