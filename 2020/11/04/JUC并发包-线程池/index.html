线程的状态

  • new 线程刚刚创建
  • ready 等待执行. 表示有执行的资格. 在等待cpu的调度
  • running 正在执行中
  • blacked 等待获取锁
  • waiting 等待 例如执行了api wait join park
  • time_waiting 有超时时间的等待
  • terminated 终止状态

线程池

通常情况下所指的线程池是指ThreadPoolExecutor. 当然还有ForkJoinPool. 线程池内部实际上构建了一个生产者消费者模型. 将线程和任务两者解耦. 通过阻塞队列对任务进行良好的缓冲. 复用线程.

ThreadPoolExecutor 基本组成

  • Worker 工作线程
  • RejectedExecutionHandler 拒绝策略
  • workQueue 工作队列
  • threadFactory 创建线程的工厂

使用线程池的好处:

  • 降低资源消耗：通过池化技术重复利用已创建的线程，降低线程创建和销毁造成的损耗。
  • 提高响应速度：任务到达时，无需等待线程创建即可立即执行。
  • 提高线程的可管理性：线程是稀缺资源，如果无限制创建，不仅会消耗系统资源，还会因为线程的不合理分布导致资源调度失衡，降低系统的稳定性。使用线程池可以进行统一的分配、调优和监控。
  • 提供更多更强大的功能：线程池具备可拓展性，允许开发人员向其中增加更多的功能。比如延时定时线程池ScheduledThreadPoolExecutor，就允许任务延期执行或定期执行。

线程池能够解决什么问题

  1. 频繁申请/销毁资源和调度资源, 带来的消耗
  2. 对资源无线申请的抑制手段. 从而减少系统资源耗尽的风险
  3. 合理管理内部资源分配. 提高系统稳定性

生命周期管理

线程池的运行状态 通过 private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0)); 字段进行管理. 高3位是runState. 低29位表示workerCount. 通过位运算维护. 共计32位. 正好一个integer.

状态有五钟:

任务执行机制

任务调度

  1. workerCount < corePoolSize 用工厂方法创建新线程执行新提交的任务
  2. workerCount >= corePoolSize && isRunning状态 && 阻塞队列未满 提交任务到阻塞队列
  3. workerCount >= corePoolSize && workerCount < maximumPoolSize && isRunning状态 && 阻塞队列满了 创建非核心线程执行
  4. workerCount >= corePoolSize && workerCount >= maximumPoolSize && isRunning状态 && 阻塞队列满了 执行拒绝策略. 默认是排除异常

非核心线程会在空间的时候 依据keepLiveTime 进行回收

任务缓存

任务缓存模块是线程池的核心模块. 线程池本质是对任务和线程的管理. 阻塞队列是一个支持两个附加操作的队列, 队列为空. 获取元素的线程会等待队列变为空. 存储元素的线程会等待队列有位置. 阻塞队列就是生产者存放元素的容器. 而消费者也只从容器里拿元素

使用不同的队列可以实现不一样的任务存取策略: 阻塞队列原理

任务申请

addWorker()有两种情况 1. 携带Task的 2. 不携带Task的. 从阻塞队列中获取任务执行

任务拒绝

任务拒绝模块是线程池的保护部分, 当线程池的任务队列已满. 线程数也达到了最大数量. 在提交任务就会执行拒绝策略

  1. AbortPolicy 默认策略 丢弃任务并抛出RejectedExecutionException异常 通过捕获异常发现问题 关键任务
  2. DiscardPolicy 丢弃任务. 不抛出异常 捕获不到任何异常也就无法发现任务呗拒绝. 非关键任务
  3. DiscardOldestPolicy 丢弃队列最前面的任务 然后重新提交被拒绝的任务. 权衡是否有必要
  4. CallerRunsPolicy 由调用线程处理该任务. 适合大量计算的任务类型

线程管理

Worker线程

Worker就是所谓的工作线程实体. 创建一个Worker就会绑定一个Thread. 单不一定会绑定FirstTast.

Worker执行任务持有firstTast的Worker, 执行自己的Task, 没有持有Task的Worker. 通过getTask()方法从阻塞队列中获取任务执行.

Worker基础了AQS. 使用State控制线程的状态. 1. Lock方法一旦获取了独占锁, 表示线程正在执行任务中. 2. 正在执行的任务是不可以中断的. 3. 如果该线程再不是独占锁状态. 也就是空闲状态. 是可以中断的. TryLock方法就是在判断线程是否是空闲状态

addWorker方法

  1. 判断线程池的状态是否可以添加任务
  2. 判断线程是否超过限制
  3. 在MainLock(重入锁)的保护下将worker添加到Workers
  4. 启动线程

ps: 如果添加核心线程执行的任务. 未能添加到Workers集合中. 会执行addWorkerFailed(w)方法 回滚刚刚发生的事情. 工作线程计数器-1, 从Workers集合中删除

runWorker方法

  1. 执行自己持有的task或者通过getTask从阻塞队列获取 2. 执行task直到getTask==null. 3. 执行processWorkerExit销毁线程

优雅的关闭

关闭线程

服务线程执行耗时任务. 调用方希望可以在任何时间点取消任务.

  1. 服务方线程使用volatile的标志成员变量. 提供方法改变变量状态. 线程每次执行之前检查volatile位. 做退出操作.
  2. 合理的处理interrupt位. 即执行前判断线程的中断标识. 抛出中断异常或者return; 如果执行过程中使用线程相关阻塞api 如果sleep join object.wait() 或者能够响应中断api. 一定要记得重新将中断位设置为true.

关闭线程池

依据线程池的状态, 可以指定 SHUTDOWN和STOP两个状态的处理逻辑是不同的.

  • 共同点是都不会接受新任务的提交
  • 不同点是会继续执行队列中的任务, 后者不会执行队列中的任务并调用中断方法区中断正在执行的任务

线程池对关闭提供了3个API:

  1. shutdown 线程池标记为SHUTDOWN状态. 不接受新任务. 中断空闲的线程.

疑问点在 下面代码片段case 1, 2, 3. 场景执行顺序 case1 case3 case2. 会中断执行的任务么? 答案是不会的. 因为 判断了线程池的状态 >=STOP.
疑问点2. 代码片段执行case 1, 2, 3. 场景执行顺序 case1 case3 case2. runable和callbable 处理了中断标记. 会打断执行么? 答案是不会. 因为 case2代码中使用Thread.interrupted()api 会清除中断状态. 所以run方法可以正常执行

  1. shutdownnow 将线程池的状态设置为STOP. 不接受新任务. 中断空闲的及其run的线程(case 3做到的) 不再执行阻塞队列中的任务

疑问点: 如何做到不再执行阻塞队列中的任务 下面的代码片段 case 1,2,3 展示了这个过程. 在从阻塞队列获取任务的时候分两种情况. 1. 执行阻塞队列get方法之前. 检查线程池的状态如果>=STOP. 返回NULL. 保证了阻塞队列中的任务不会被执行 2. 在阻塞队列的等待的线程在等待过程中被中断了, 因为捕获了中断异常+死循环 所以还是能返回null. 保证了阻塞队列中的任务不会被执行
疑问点: 如何做到中断正在运行RUN方法的线程. ** 这个依赖于开发程序的人. 在run方法中正确处理线程的中断位置及其中断异常 **
